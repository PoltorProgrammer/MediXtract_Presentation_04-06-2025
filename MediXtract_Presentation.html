🌍 Overall Vision

We aim to build a modular, intelligent system that automates the extraction of structured patient data from unstructured clinical documents (mostly PDFs and images), validates it, stores it safely, and presents it in accessible, editable formats (like Excel).

This system is designed to:

Run offline on a laptop traveling between hospitals
Reduce human workload and errors
Continuously improve over time through human feedback
Scale later to centralized or cloud-based systems if needed
Respect sensitive data handling and privacy

System Architecture

We break the system into four main layers, each modular and replaceable.

Input Layer

Source documents:
  • Patient PDFs, scanned images, lab reports, discharge summaries
Preprocessing:
  • If PDF contains text → parse directly
  • If PDF/image contains only image → apply OCR (Optical Character Recognition) (e.g., Tesseract or commercial solutions like Google Vision API)
Normalization:
  • Standardize encodings, handle noisy scans, correct rotations, apply image enhancements if necessary

Extraction Layer

Core component: Multi-AI agent system, where each agent specializes in:
  • Document understanding (layout, section detection)
  • Named entity recognition (detecting patient names, dates, measurements, diagnoses)
  • Relationship extraction (linking, e.g., lab result X → date Y → patient Z)
AI models:
  • Use foundation models (Claude, GPT-4o, Gemini) via API to accelerate extraction
  • For highly specialized terms (medical jargon), optionally fine-tune smaller domain-specific models or add rule-based patterns
Caching & optimization:
  • Reduce token/API costs by caching reusable prompt components
  • Identify "frequent patterns" (e.g., lab report templates) and treat them with lightweight local models to save costs

Data Layer

Temporary data store:
  • MongoDB (local, document-based DB, good for semi-structured data)
Validation mechanisms:
  • Schema validation using tools like Pydantic (define expected formats, ranges, types)
  • Outlier detection: flag unusual values (e.g., impossible lab values) for human review
  • Confidence scores: attach uncertainty measures to each extracted field
Audit & logging:
  • Keep traceable logs of all extraction steps, making sure we can trace back any mistake to its origin

Output & User Layer

Excel generation:
  • Automatically generate structured Excel files with correct column mapping
  • Use libraries like pandas + openpyxl (Python)
User interface (UI):
  • Built in Ruby on Rails (or a lightweight alternative like Flask/FastAPI + React)
  • Key functions: upload documents, monitor progress, review flagged entries, download results
  • Provide an admin dashboard showing statistics, error rates, token usage, and system performance

📊 Budget Considerations

Main cost drivers

  • AI API usage (Claude, Gemini, GPT) → charged per token processed

Advantages of this budget model

  • We only pay based on actual use — no need to invest in costly server infrastructure upfront

  • Pay-as-you-go model reduces financial risk
  • Initial investment primarily in development, not infrastructure

🌍 Gesamtvision

Wir zielen darauf ab, ein modulares, intelligentes System zu entwickeln, das die Extraktion strukturierter Patientendaten aus unstrukturierten klinischen Dokumenten (hauptsächlich PDFs und Bilder) automatisiert, validiert, sicher speichert und in zugänglichen, bearbeitbaren Formaten (wie Excel) präsentiert.

Dieses System ist darauf ausgelegt:

Offline auf einem Laptop zu laufen, der zwischen Krankenhäusern reist
Menschliche Arbeitsbelastung und Fehler zu reduzieren
Sich kontinuierlich durch menschliches Feedback zu verbessern
Später auf zentralisierte oder cloud-basierte Systeme zu skalieren, falls nötig
Sensible Datenverarbeitung und Datenschutz zu respektieren

Systemarchitektur

Wir unterteilen das System in vier Hauptschichten, jede modular und austauschbar.

Eingabeebene

Quelldokumente:
  • Patienten-PDFs, gescannte Bilder, Laborberichte, Entlassungsberichte
Vorverarbeitung:
  • Falls PDF Text enthält → direkt parsen
  • Falls PDF/Bild nur Bild enthält → OCR (Optische Zeichenerkennung) anwenden (z.B. Tesseract oder kommerzielle Lösungen wie Google Vision API)
Normalisierung:
  • Kodierungen standardisieren, verrauschte Scans behandeln, Rotationen korrigieren, Bildverbesserungen anwenden falls nötig

Extraktionsebene

Kernkomponente: Multi-KI-Agentensystem, wobei jeder Agent sich spezialisiert auf:
  • Dokumentverständnis (Layout, Sektionserkennung)
  • Named Entity Recognition (Erkennung von Patientennamen, Daten, Messungen, Diagnosen)
  • Beziehungsextraktion (Verknüpfung, z.B. Laborergebnis X → Datum Y → Patient Z)
KI-Modelle:
  • Nutzung von Foundation-Modellen (Claude, GPT-4o, Gemini) via API zur Beschleunigung der Extraktion
  • Für hochspezialisierte Begriffe (medizinischer Jargon), optional Fine-Tuning kleinerer domänenspezifischer Modelle oder regelbasierte Muster hinzufügen
Caching & Optimierung:
  • Token/API-Kosten reduzieren durch Caching wiederverwendbarer Prompt-Komponenten
  • "Häufige Muster" identifizieren (z.B. Laborberichtvorlagen) und mit leichtgewichtigen lokalen Modellen behandeln, um Kosten zu sparen

Datenebene

Temporärer Datenspeicher:
  • MongoDB (lokal, dokumentbasierte DB, gut für semi-strukturierte Daten)
Validierungsmechanismen:
  • Schema-Validierung mit Tools wie Pydantic (erwartete Formate, Bereiche, Typen definieren)
  • Ausreißererkennung: ungewöhnliche Werte (z.B. unmögliche Laborwerte) für menschliche Überprüfung markieren
  • Vertrauenswerte: Unsicherheitsmaße an jedes extrahierte Feld anhängen
Audit & Protokollierung:
  • Nachverfolgbare Protokolle aller Extraktionsschritte führen, um sicherzustellen, dass wir jeden Fehler zu seinem Ursprung zurückverfolgen können

Ausgabe- & Benutzerebene

Excel-Generierung:
  • Automatische Generierung strukturierter Excel-Dateien mit korrekter Spaltenzuordnung
  • Verwendung von Bibliotheken wie pandas + openpyxl (Python)
Benutzeroberfläche (UI):
  • Entwickelt in Ruby on Rails (oder einer leichtgewichtigen Alternative wie Flask/FastAPI + React)
  • Hauptfunktionen: Dokumente hochladen, Fortschritt überwachen, markierte Einträge überprüfen, Ergebnisse herunterladen
  • Admin-Dashboard mit Statistiken, Fehlerquoten, Token-Verbrauch und Systemleistung bereitstellen
