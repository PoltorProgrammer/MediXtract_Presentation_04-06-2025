üåç Overall Vision

We aim to build a modular, intelligent system that automates the extraction of structured patient data from unstructured clinical documents (mostly PDFs and images), validates it, stores it safely, and presents it in accessible, editable formats (like Excel).

This system is designed to:

Run offline on a laptop traveling between hospitals
Reduce human workload and errors
Continuously improve over time through human feedback
Scale later to centralized or cloud-based systems if needed
Respect sensitive data handling and privacy

System Architecture

We break the system into four main layers, each modular and replaceable.

Input Layer

Source documents:
  • Patient PDFs, scanned images, lab reports, discharge summaries
Preprocessing:
  • If PDF contains text ‚Üí parse directly
  • If PDF/image contains only image ‚Üí apply OCR (Optical Character Recognition) (e.g., Tesseract or commercial solutions like Google Vision API)
Normalization:
  • Standardize encodings, handle noisy scans, correct rotations, apply image enhancements if necessary

Extraction Layer

Core component: Multi-AI agent system, where each agent specializes in:
  • Document understanding (layout, section detection)
  • Named entity recognition (detecting patient names, dates, measurements, diagnoses)
  • Relationship extraction (linking, e.g., lab result X ‚Üí date Y ‚Üí patient Z)
AI models:
  • Use foundation models (Claude, GPT-4o, Gemini) via API to accelerate extraction
  • For highly specialized terms (medical jargon), optionally fine-tune smaller domain-specific models or add rule-based patterns
Caching & optimization:
  • Reduce token/API costs by caching reusable prompt components
  • Identify "frequent patterns" (e.g., lab report templates) and treat them with lightweight local models to save costs

Data Layer

Temporary data store:
  • MongoDB (local, document-based DB, good for semi-structured data)
Validation mechanisms:
  • Schema validation using tools like Pydantic (define expected formats, ranges, types)
  • Outlier detection: flag unusual values (e.g., impossible lab values) for human review
  • Confidence scores: attach uncertainty measures to each extracted field
Audit & logging:
  • Keep traceable logs of all extraction steps, making sure we can trace back any mistake to its origin

Output & User Layer

Excel generation:
  • Automatically generate structured Excel files with correct column mapping
  • Use libraries like pandas + openpyxl (Python)
User interface (UI):
  • Built in Ruby on Rails (or a lightweight alternative like Flask/FastAPI + React)
  • Key functions: upload documents, monitor progress, review flagged entries, download results
  • Provide an admin dashboard showing statistics, error rates, token usage, and system performance

Cost Optimization Strategies

Several strategies can dramatically reduce processing costs by optimizing how and when resources are used:

  • Batch API Processing: Group API calls into scheduled batches rather than sending them individually. This approach qualifies for volume discounts (up to 50% off Claude API costs), making it highly efficient for large-scale operations.
  • Template Recognition: Detect and cache the structure of frequently used document formats (e.g., invoices or reports). By reusing these templates, the system avoids redundant parsing, reducing token usage and accelerating processing.
  • Gradual Processing: Distribute the workload of retrospective document analysis over 12‚Äì24 months. This avoids sudden spikes in cost and aligns processing with budget cycles, enabling sustainable long-term scaling.
  • Smart OCR: Apply a tiered OCR strategy. Use free tools like Tesseract for clean, simple documents. Only utilize paid OCR solutions (e.g., Google Vision or ABBYY) when higher accuracy or advanced features (like table extraction or handwriting recognition) are required.

By combining these methods, we ensure high performance and scalability while keeping costs under control.
